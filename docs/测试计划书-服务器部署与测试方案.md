## 1. 部署架构

### 1.1 核心决策（已确认）

> ✅ **服务器中心化**：所有配置、Bot 进程、Vault 数据均在服务器端运行和存储。用户无需本地部署 Bot，仅通过浏览器访问前端网站即可使用全部功能。

### 1.2 架构全景

```
┌──────────────────────────────────────────────────────────┐
│                      用户浏览器                             │
│           https://cyberboss.example.com                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌─────────────┐ │
│  │ 扫码登录  │ │ .env配置  │ │ 用量看板  │ │  API Explorer│ │
```

### 1.3 已确认的技术方向

| 层面 | 决定 | 说明 |
|------|------|------|
| **部署模型** | 服务器中心化 | 用户不需要本地部署 Bot |
| **配置方式** | 前端表单 → 服务端存储 | 用户填写表单，服务器写库 + 生成 `.env` |
| **登录方式** | 二维码扫码登录 | Web 展示二维码 → 微信扫码 → 服务器确认 |
| **Vault 存储** | 服务器存储 | 每个用户独立的 vault 目录，服务器端管理 |
| **Vault 同步** | 双向同步 | 服务器 Vault ↔ 用户本地 Vault |

### 1.4 待决策

> **🔴 决策 1.1：前端框架选择**
>
> - **React + Vite + shadcn/ui**：生态最大，组件丰富，适合管理后台类应用
> - **Vue 3 + Vite + Naive UI**：中文社区好，上手快
> - **纯 HTML + Alpine.js / HTMX**：轻量但缺乏组件化
>
> **建议**：React + Vite + shadcn/ui。

> **🔴 决策 1.2：前端与后端服务的组织方式**
>
> - **方案 A（单仓库 Monolith）**：前端 build 产物放入 `cyberboss/public/`，由现有 Express 静态托管。简单，单进程。
> - **方案 B（独立前端服务）**：前端独立 Vite dev server / Nginx，通过 `/api` 反代到后端。解耦，开发热更新方便。
>
> **建议**：开发期用方案 B（Vite dev proxy），生产构建用方案 A（Express 静态托管单进程部署）。

> **🔴 决策 1.3：多用户 Bot 进程管理**
>
> - **方案 A（PM2 多进程）**：每个用户一个 Node 子进程跑 Bot，PM2 管理生命周期
> - **方案 B（Docker 容器化）**：每个用户一个容器，资源隔离最好但开销大
> - **方案 C（单进程多实例）**：一个 Node 进程内管理多个 Bot 实例，共享内存但隔离性弱
>
> **建议**：方案 C（单进程多实例）MVP 起步。每个用户一个 Bot 实例对象，共享 MCP Tool Host。资源利用率高，实现简单。后续按需迁移到 Docker。

## 2. 环境变量可视化配置

### 2.1 核心决策（已确认）

> ✅ **前端表单填写 + 服务端存储 + 服务端生效**。用户在前端填写配置 → 服务端校验、加密存储、生成 `.env` → Bot 进程读取 `.env` 启动。用户不需要接触 `.env` 文件。

### 2.2 配置流程

```
用户浏览器                    服务器
    │                          │
    │  GET /api/config          │
    │ ─────────────────────►   │ 读取用户当前配置（脱敏）
    │  返回当前配置表单          │
    │ ◄─────────────────────   │
    │                          │
    │  填写/修改配置             │
    │  POST /api/config         │
    │ ─────────────────────►   │ 校验 + 加密 API Key
    │                          │ → 写入 SQLite
    │                          │ → 生成 .env 文件
    │                          │ → 重启 Bot（或热加载）
    │  返回 { ok: true }       │
    │ ◄─────────────────────   │
    │                          │
    │  POST /api/config/test    │
    │ ─────────────────────►   │ 用当前配置调一次 API
    │  返回 { ok, latency }    │  → 测试连通性
    │ ◄─────────────────────   │
```

### 2.3 配置项清单

```env
# === 必填（用户在前端填写） ===
CYBERBOSS_USER_NAME=            # 文本输入 + 占位提示：你的名字
CYBERBOSS_ALLOWED_USER_IDS=     # 文本输入 + 提示：微信 wxid
CYBERBOSS_RUNTIME=              # 下拉选择：claudecode（默认）
CYBERBOSS_ACCOUNT_ID=           # 文本输入（微信机器人 ID）

# === 模型配置 ===
CYBERBOSS_MODEL_PROVIDER=       # 下拉：anthropic | openai | deepseek | custom
CYBERBOSS_MODEL_NAME=           # 下拉 + 自定义
CYBERBOSS_API_KEY=              # 密码输入 + 模板提示：sk-xxxx
CYBERBOSS_API_BASE_URL=         # 文本 + 根据 provider 自动填充默认值

# === 服务器自动填充（用户不可见） ===
CYBERBOSS_WORKSPACE_ROOT=       # 服务器分配的用户工作目录
CYBERBOSS_OBSIDIAN_VAULT=       # 服务器分配的用户 vault 路径
CYBERBOSS_TZ=                   # 用户选择 → 服务端写入
CYBERBOSS_DAILY_SUMMARY_TIME=   # 时间选择器
```

### 2.4 安全设计

- API Key 用 AES-256-GCM 加密后存入 SQLite，`.env` 中可存明文（Bot 运行时需要）
- 前端 API 返回配置时，API Key 字段永远脱敏（`sk-****xxxx`）
- `.env` 文件权限 `600`，用户 vault 目录权限隔离
- 配置变更记录审计日志（时间、字段、IP）

### 2.5 待决策

> **🔴 决策 2.1：配置存储方案**
>
> - **方案 A（SQLite 主存储 + .env 导出）**：配置存在 SQLite，每次变更时重新生成 `.env`。支持历史版本、回滚。
> - **方案 B（纯 .env 文件）**：直接读写 `.env`。简单但无历史、无校验。
>
> **建议**：方案 A。SQLite 做主存储，`.env` 是 Bot 运行时的消费格式。

> **🟡 决策 2.2：配置变更后 Bot 是否需要重启？**
>
> - **热加载**：Bot 监听配置变更信号，无需重启。但实现复杂（尤其 Claude Code 进程）。
> - **自动重启**：配置变更后自动重启用户 Bot 进程。简单可靠，但有 5-10 秒中断。
>
> **建议**：自动重启。测试阶段可接受短暂中断，MVP 不引入热加载复杂度。

> **🟡 决策 2.3：模型提供商的选项来源**
>
> - **静态列表**：硬编码常见提供商
> - **动态模板**：`provider-templates.json` 维护默认 Base URL + 模型列表，支持从 GitHub 更新
>
> **建议**：静态列表 + 动态模板。首次静态加载，后台拉取最新模板。

## 3. 用户仓库管理与双向同步

### 3.1 核心决策（已确认）

> ✅ **服务器存储 Vault + 双向同步**。
> - 服务器端：每个用户拥有独立的 Obsidian Vault 目录，Bot 进程直接读写
> - 用户本地：用户可拥有自己的 Vault 副本（Obsidian 打开编辑）
> - 双向同步：服务器端变更 → 用户本地；用户本地变更 → 服务器端

### 3.2 数据模型

```
服务器 /data/vaults/
├── user_wxid_001/              # 用户 A 的 vault（隔离）
│   ├── 闪存/
│   ├── 日记/
│   ├── 大构思/
│   ├── 人际关系/
│   ├── 知识库/
│   └── .cyberboss/
│       ├── config.sqlite       # 用户配置
│       └── usage.sqlite        # 用量数据
│
├── user_wxid_002/              # 用户 B 的 vault（隔离）
│   └── ...
```

### 3.3 双向同步方案

```
┌─────────────────────────────────────────┐
│              服务器 Vault                 │
│  /data/vaults/user_wxid_001/            │
│  (Bot 直接读写，MCP Tools 操作)            │
└──────────┬──────────────────────────────┘
           │
     ┌─────▼────── 双向同步 ──────┐
     │                            │
     ▼                            ▼
┌─────────────┐          ┌──────────────┐
│ Git 方式     │          │ 文件同步方式   │
│ (推荐 MVP)   │          │ (后续可选)     │
├─────────────┤          ├──────────────┤
│· git pull   │          │· rsync       │
│· git push   │          │· Syncthing   │
│· 冲突标记    │          │· WebDAV      │
│· .gitignore │          │· Obsidian    │
│  排除大文件  │          │  Sync API    │
└──────┬──────┘          └──────┬───────┘
       │                        │
       └────────┬───────────────┘
                ▼
     ┌─────────────────────┐
     │   用户本地 Vault      │
     │  ~/Obsidian/cyberboss│
     │  (Obsidian 打开编辑)  │
     └─────────────────────┘
```

### 3.4 Git 双向同步流程（推荐 MVP 方案）

```
服务器端变更（Bot 写入了闪存/日记/总结等）
  │
  ├─→ 变更事件触发（或定时批量）
  ├─→ git add -A
  ├─→ git commit -m "cyberboss: auto-sync $(date)"
  ├─→ git pull --rebase origin main    # 先拉用户本地变更
  │     ├─ 无冲突 → 继续
  │     └─ 有冲突 → 标记冲突文件 + 通知用户
  ├─→ git push origin main
  └─→ WebSocket/轮询 通知用户"Vault 已同步"

用户本地变更（用户用 Obsidian 编辑了文件）
  │
  ├─→ 用户手动 git commit + push（或 Obsidian Git 插件自动）
  ├─→ 服务器定时 git pull（或 webhook 触发）
  ├─→ 检测到变更 → 触发相关 MCP Tool 重新索引
  └─→ 通知 Bot"Vault 有更新"
```

### 3.5 冲突处理策略

| 冲突场景 | 处理方式 |
|----------|---------|
| 同一文件、不同行 | Git 自动合并 |
| 同一文件、同一行 | 保留双方版本（`<<<<<<` 标记），通知用户手动解决 |
| 文件删除 vs 修改 | 保留修改版本，通知用户 |
| 二进制文件冲突 | 保留服务器版本，用户本地版本备份为 `.conflict` |

### 3.6 待决策

> **🔴 决策 3.1：双向同步的实现方式**
>
> - **方案 A（Git）**：每个用户 vault 是一个 Git 仓库。服务器和用户本地通过 `git pull/push` 双向同步。冲突有成熟处理机制。
> - **方案 B（rsync / Syncthing）**：文件级双向同步，实时性好，但冲突处理弱，需要额外 daemon。
> - **方案 C（Obsidian Sync API）**：官方同步服务，最可靠但需要用户购买 Obsidian Sync 订阅。
> - **方案 D（WebDAV 挂载）**：用户 vault 直接挂载到服务器文件系统。无需同步但网络延迟影响 Bot 性能。
>
> **建议**：方案 A（Git）MVP 起步。项目已有 Git 操作能力，冲突处理成熟。后续可加 Syncthing 做实时同步。

> **🔴 决策 3.2：同步触发时机**
>
> - **定时批量**：每 N 分钟（如 5 分钟）自动 git pull + push。简单，有延迟。
> - **事件驱动**：MCP Tool 写操作后立即触发 git push；webhook 收到用户 push 后立即 git pull。
> - **混合模式**：服务器写 → 立即 push（事件驱动）；用户本地变更 → 定时 pull（每 5 分钟）+ 用户手动点"同步"按钮。
>
> **建议**：混合模式。服务器端事件驱动 push，用户端定时 pull + 手动同步按钮。

> **🟡 决策 3.3：Vault 数据安全**
>
> - **磁盘加密**：服务器上 vault 目录用 LUKS / BitLocker 加密
> - **传输加密**：Git push/pull 走 SSH（不用 HTTPS + token）
> - **访问控制**：每个用户 vault 目录权限隔离（Linux 用户隔离 / Docker 容器隔离）
> - **密钥管理**：用户 SSH key 在前端上传，服务器不存明文
>
> **建议**：MVP 先做目录权限隔离 + SSH 传输。磁盘加密和密钥管理后续加强。

> **🟡 决策 3.4：服务器为用户分配 Git 仓库还是使用用户自有仓库？**
>
> - **方案 A（服务器分配）**：服务器给每个用户创建私有 Git 仓库（Gitea / GitLab 自建），用户 clone 到本地
> - **方案 B（用户自有）**：用户提供自己的 GitHub/GitLab 仓库地址，服务器 clone 并同步
>
> **建议**：方案 B（用户自有仓库）。用户数据所有权归用户，服务器只是同步节点。

## 4. 报告同步与工单系统

### 4.1 核心决策（已确认）

> ✅ **服务器端工单系统**：测试者反馈自动进入工单系统，开发者可在前端查看/分配/回复。由于 Bot 已在服务器运行，反馈直接写入服务器工单 DB，无需"上报"环节。

### 4.2 反馈流转（简化后）

```
测试者微信 ──→ Bot (服务器内) ──→ MCP Tool
                                    │
                    ┌───────────────┤
                    ▼               ▼
              用户 vault       工单系统 (同服务器 DB)
              (完整版)         (开发者可见)
                                    │
                    ┌───────────────┤
                    ▼               ▼
              开发者查看/回复   通知测试者（微信 + Web）
```

### 4.3 工单数据模型

```typescript
interface Ticket {
  id: string;                    // TK-20260626-001
  userId: string;                // 关联用户
  type: 'bug' | 'feature' | 'ux' | 'other';
  title: string;
  content: string;               // 原始反馈内容
  priority: 'high' | 'medium' | 'low';
  status: 'open' | 'acknowledged' | 'in_progress' | 'resolved' | 'closed';
  assignee: string;              // 开发者
  tags: string[];
  source: string;                // 来源：wechat / web
  createdAt: string;
  updatedAt: string;
  comments: {
    author: string;
    content: string;
    createdAt: string;
    visibility: 'tester' | 'developer' | 'both';  // 可见性
  }[];
}
```

### 4.4 待决策

> **🔴 决策 4.1：工单系统实现方式**
>
> - **方案 A（自建）**：在服务器内置工单系统（SQLite + 前端管理页）。完全可控，与系统深度集成。
> - **方案 B（GitHub Issues）**：反馈自动创建 Issue。零开发前端，但测试者无法直接看到进度。
> - **方案 C（飞书/企微多维表格）**：反馈写入多维表格。协作方便但依赖第三方。
>
> **建议**：方案 A（自建）。服务器架构下自建成本低，Bot + 工单在同一进程，反馈延迟为零。

> **🟡 决策 4.2：开发者回复的通知渠道**
>
> - 开发者在工单系统中回复 → 通过 Bot 发微信消息给测试者
> - 或者 → 测试者下次打开 Web 面板时看到回复
>
> **建议**：双通道。紧急回复走微信，普通回复在 Web 面板展示。

## 5. 前端功能调用示例

### 5.1 目标

网站前端提供**全部功能**的调用示例（API 请求演示、参数说明、返回结果展示）。

### 5.2 方案设计

由于 Bot 运行在服务器端，前端可以直接通过服务器 API 调 Bot 功能，**不需要 Mock**：

```
┌────────────────────────────────────────────┐
│  🔌 API Explorer                           │
│                                            │
│  ┌─ 闪存记忆 ─────────────────────────────┐ │
│  │ POST /api/user/flash-memory/capture    │ │
│  │   📥 参数 (自动填充当前用户)              │ │
│  │   ┌──────────────────────────────────┐ │ │
│  │   │ {                                │ │ │
│  │   │   "text": "在通勤路上想到一个主意",  │ │ │
│  │   │   "category": "idea",            │ │ │
│  │   │   "tags": ["灵感"],              │ │ │
│  │   │   "priority": "medium"           │ │ │
│  │   │ }                                │ │ │
```

## 9. 技术难点专项分析

> 本章针对小方提出的三个关键技术难点进行专项分析，每个难点给出可行的技术方案和对比评估。

### 9.1 扫码登录 Web 化

#### 9.1.1 问题描述

原版 Cyberboss 的 Bot 启动流程：`powershell` 启动 Bot → WeChat SDK 生成登录二维码 → 二维码打印到终端窗口 → 用户用微信扫码 → 登录成功。

在 Web 端，没有终端可供打印。核心问题是：**如何截获 WeChat SDK 生成的二维码图片，并将其传递到浏览器展示？**

#### 9.1.2 原版流程分析

```
WeChat SDK (wechaty / 自定义)
  │
  ├─→ 调用微信官方 API 获取登录二维码
  │     └─→ 返回：二维码图片 (Buffer/URL) 或 二维码数据 (base64/qrcode terminal)
  │
  ├─→ 将二维码输出到 stdout/stderr
  │     └─→ 原版在 PowerShell 中显示为终端二维码（ASCII art 或图片渲染）
  │
  └─→ 轮询登录状态 → 登录成功 → Bot 上线
```

#### 9.1.3 Web 化方案

**核心思路**：不用终端输出，改为捕获二维码数据 → 存储为图片文件 → HTTP/WebSocket 推送前端。

```
┌─────────────────────────────────────────────────────────┐
│  服务器端                                                  │
│                                                           │
│  Bot 启动流程（改造后）                                      │
│    │                                                      │
│    ├─→ 1. 调用 WeChat SDK → 获取二维码                      │
│    │      └─→ 方案 A: SDK 返回 QR 图片 URL                 │
│    │      └─→ 方案 B: SDK 返回 QR 数据 Buffer              │
│    │      └─→ 方案 C: SDK 打印到 stdout，hook 拦截          │
│    │                                                      │
│    ├─→ 2. 将二维码保存到用户目录                             │
│    │      /data/users/{userId}/tmp/qrcode.png              │
│    │                                                      │
│    ├─→ 3. 推送给前端                                        │
│    │      ├─→ WebSocket: { event: "qrcode", data: "<base64>" }│
│    │      └─→ HTTP GET /api/user/qrcode → 返回图片          │
│    │                                                      │
│    └─→ 4. 前端展示 <img src="data:image/png;base64,...">   │
│                                                           │
└─────────────────────────────────────────────────────────┘
```

#### 9.1.4 三种拦截方案对比

| 方案 | 原理 | 改动量 | 可靠性 | 延迟 |
|------|------|--------|--------|------|
| **A. SDK API 直接取** | 如果 WeChat SDK 提供了获取二维码 Buffer 的 API，直接调用拿到图片数据 | 最小 | ⭐⭐⭐ 最高 | 最低 |
| **B. Hook stdout 拦截** | Bot 进程的 stdout 被重定向，正则匹配二维码内容（ASCII 或 base64），提取出来 | 中等 | ⭐⭐ 中等 | 低 |
| **C. 文件系统监听** | SDK 可能将二维码保存为临时文件 → 监听文件系统变化 → 读取 → 推送 | 较小 | ⭐⭐ 中等 | 低 |

#### 9.1.5 推荐实施方案

```
                         用户浏览器
                              │
                    ① 点击"启动 Bot"
                              │
                    ② POST /api/user/bot/start
                              │
                              ▼
                         ┌──────────┐
                         │ 服务器     │
                         │          │
                         │ ③ 启动Bot │
                         │   子进程   │
                         │          │
                         │ ④ Hook   │
                         │   stdout  │──→ 检测到二维码 ──→ 保存 qrcode.png
                         │          │
                         │ ⑤ WebSocket emit             │
                         │   { qrcode: "/tmp/xxx.png" } │
                         └──────────┘
                              │
                    ⑥ GET /api/user/qrcode
                    ⑦ 返回二维码图片 (PNG)
                              │
                              ▼
                         ┌──────────┐
                         │ 前端      │
                         │ <img>    │
                         │ 展示二维码 │
                         │          │
                         │ 轮询登录态 │──→ ⑧ GET /api/user/bot/status
                         │          │     返回 { status: "scanning" }
                         └──────────┘     返回 { status: "online" }
```

**推荐策略**：
1. **优先方案 A**：先检查使用的 WeChat SDK 是否有直接获取二维码的 API（大部分 SDK 都有 `on('scan', (qrcode) => {...})` 事件，其中的 `qrcode` 参数就是二维码 URL 或 base64）
2. **兜底方案 B**：如果 SDK 只打印到终端，Hook 子进程 stdout，用正则提取二维码数据
3. **前端展示**：WebSocket 推送 + HTTP 兜底轮询，`<img>` 直接渲染

#### 9.1.6 待决策

> **🔴 决策 9.1：扫码登录 Web 化采用哪种拦截方案？**
>
> - **方案 A（SDK API 直接取）**：优先尝试，改动最小、最可靠。需确认 WeChat SDK 的具体 API。
> - **方案 B（stdout Hook）**：兜底方案，适用于所有情况，但需要处理终端控制字符。
>
> **建议**：方案 A 优先，方案 B 兜底。先花 1 小时调研当前使用的 WeChat SDK 的二维码事件 API，如果有直接回调则用 A，否则用 B。

### 9.2 服务器端截图

#### 9.2.1 问题描述

每日总结、周/月总结、时间轴的截图功能依赖 Puppeteer（Headless Chrome）渲染 HTML 模板并截图。在 VPS（Linux 无桌面环境）上是否难以实现？

#### 9.2.2 结论：服务器端截图比桌面端更简单

Puppeteer 的**标准运行环境就是 Linux 无桌面服务器**。原版在 Windows 桌面遇到的一切问题（字体渲染、PowerShell 调用、显示驱动），在 Linux VPS 上反而不存在。

| 对比维度 | Windows 桌面（现在） | Linux VPS（目标） |
|----------|---------------------|-------------------|
| Puppeteer 运行 | 需要找到本地 Chrome 路径 | `npm i puppeteer` 自动下载 Chromium |
| 截图稳定性 | 依赖桌面环境 | Headless 模式原生支持 |
| 中文渲染 | 系统自带中文字体 | 需要安装 `fonts-noto-cjk`（一行命令） |
| 内存占用 | ~300MB / Chrome 实例 | ~200MB / Chromium 实例 |
| GPU 依赖 | 偶有 GPU 渲染问题 | 纯 CPU 渲染，无 GPU 问题 |
| 并发截图 | 受限于桌面资源 | 可管理 Chromium 进程池 |

#### 9.2.3 服务器端截图技术方案

```
┌─────────────────────────────────────────────────────────┐
│  截图请求（任一来源）                                       │
│    ├─ 定时任务：22:57 每日总结                              │
│    ├─ 手动触发：用户微信发 "/summary"                       │
│    ├─ Web 面板：用户点击 "生成截图"                         │
│    └─ 周/月总结：周日 / 每月 15 号                          │
│                                                           │
│              ▼                                            │
│  ┌──────────────────────────────────────┐                │
│  │        ScreenshotService              │                │
│  │                                       │                │
│  │  1. 渲染 HTML 模板                     │                │
│  │     ├─ 读模板：templates/daily.html    │                │
│  │     ├─ 注入数据：从 vault + DB 聚合     │                │
│  │     └─ 生成完整 HTML（已内联 CSS）       │                │
│  │                                       │                │
│  │  2. Puppeteer 截图                     │                │
│  │     ├─ 启动 headless Chromium          │                │
│  │     ├─ page.goto(file://...) 加载 HTML  │                │
│  │     ├─ page.screenshot({ fullPage })   │                │
│  │     └─ 关闭页面（复用 browser 实例）      │                │
│  │                                       │                │
│  │  3. 输出                               │                │
│  │     ├─ 保存 PNG → vault 截图目录         │                │
│  │     ├─ 通过 Bot 发送到微信               │                │
│  │     └─ 返回 URL 给 Web 前端展示          │                │
│  └──────────────────────────────────────┘                │
└─────────────────────────────────────────────────────────┘
```

#### 9.2.4 VPS 环境依赖清单

```bash
# 1. Puppeteer + Chromium（npm 自动安装）
npm install puppeteer

# 2. Chromium 系统依赖（Debian/Ubuntu）
apt install -y \
  ca-certificates fonts-liberation libasound2 \
  libatk-bridge2.0-0 libatk1.0-0 libcups2 \
  libdrm2 libgbm1 libgtk-3-0 libnspr4 \
  libnss3 libu2f-udev libxcomposite1 \
  libxdamage1 libxfixes3 libxkbcommon0 \
  libxrandr2 xdg-utils

# 3. 中文字体（关键！）
apt install -y fonts-noto-cjk

# 4. 验证
node -e "require('puppeteer').launch({headless:true}).then(b=>b.newPage().then(p=>p.screenshot({path:'test.png'}).then(()=>{console.log('OK');b.close()})))"
```

#### 9.2.5 性能与资源评估

| 场景 | Chromium 内存 | 截图耗时 | 并发建议 |
|------|-------------|---------|---------|
| 单次每日总结 | ~150MB | ~2-3s | — |
| 周总结（7天数据） | ~200MB | ~4-5s | — |
| 时间轴日视图 | ~150MB | ~2s | — |
| 多用户同时截图 | 每实例 ~150MB | 每任务 ~3s | Chromium 进程池（3-5 个实例复用） |

**关键优化**：
- Browser instance 复用（不每次 new，用 `puppeteer.connect()` 连到已启动的 browser）
- 截图队列：同时只跑 N 个截图任务，其余排队
- 低优先级：每日总结 22:57 集中触发时按用户错峰 30s

#### 9.2.6 与现有 Screenshot MCP Tool 的兼容性

现有 MCP Tool（`cyberboss_summary_screenshot`、`cyberboss_timeline_screenshot`）已经调 Puppeteer，接口不变。改造重点是把 Puppeteer 的启动方式从"找系统 Chrome"改为"用 puppeteer 自带的 Chromium"。

```javascript
// 现有（Windows 桌面）
const browser = await puppeteer.launch({
  executablePath: 'C:\\Program Files\\Google\\Chrome\\Application\\chrome.exe',
  headless: true,
});

// 改造后（Linux VPS）
const browser = await puppeteer.launch({
  headless: 'new',  // 或 'shell'（ puppeteer 自带 Chromium）
  args: ['--no-sandbox', '--disable-setuid-sandbox'],
});
```

> **🟡 决策 9.2：Chromium 实例管理策略**
>
> - **方案 A（单实例复用）**：一个 Chromium 实例服务所有用户。内存省，但如果崩溃影响全局。
> - **方案 B（进程池）**：预启动 3-5 个 Chromium 实例，任务分配。更稳定，内存多占 ~500MB。
> - **方案 C（按需启停）**：每次截图启动 + 用完关闭。内存最省，但每次启动 ~1s 额外延迟。
>
> **建议**：方案 B（进程池 3 实例）。VPS 1GB+ 内存足够，稳定性和响应速度最佳。

> **🟢 决策 9.3：截图文件存储位置**
>
> - 截图保存到用户 vault 的 `截图/` 目录 → 随 vault 双向同步到用户本地
> - 或保存到 `/data/screenshots/` 独立目录，仅服务器可访问
>
> **建议**：保存到 vault 内，随 Git 同步。用户本地 Obsidian 也能看到截图。

#### 9.3.1 问题描述

#### 9.3.2 需要部署的组件清单

```
Cyberboss 服务器
├── ① Node.js 运行时 (v18+)

│     ├── templates/ (HTML 模板: 每日总结、时间轴...)
```

#### 9.3.3 三种打包部署方案

| 方案 | 描述 | 适合场景 |
|------|------|---------|
| **A. Docker Compose** | 一个 `docker-compose.yml`，`docker compose up -d` 一键启动 | ⭐⭐⭐ 推荐 |
| **B. 安装脚本 + PM2** | `install.sh` 装依赖 → PM2 管理进程 | ⭐⭐ 备选 |
| **C. 手动部署文档** | 写部署文档，手动一步步操作 | ⭐ 应急 |

#### 9.3.4 推荐方案：Docker Compose

```
cyberboss-server/
├── docker-compose.yml          # 一键启动
├── Dockerfile                  # Cyberboss 主镜像
├── nginx/
│   ├── nginx.conf              # 反代配置
│   └── ssl/                    # HTTPS 证书
├── scripts/
│   ├── setup.sh                # 初始化脚本
│   └── backup.sh               # 数据备份脚本
├── data/                       # 持久化数据（挂载到容器外）
│   ├── vaults/                 # 用户 Vault
│   └── config/                 # SQLite 配置
└── .env.server                 # 服务器全局环境变量
```

**`docker-compose.yml` 结构**：

```yaml
version: '3.8'
services:
  cyberboss:
    build: .
    container_name: cyberboss-server
    restart: unless-stopped
    ports:
      - '127.0.0.1:3000:3000'       # 仅本地回环，通过 Nginx 暴露
    volumes:
      - ./data:/data                  # 持久化用户数据
      - ./templates:/app/templates    # HTML 模板
      - /var/run/docker.sock:/var/run/docker.sock  # 可选：管理子容器
    environment:
      - NODE_ENV=production
      - DATA_ROOT=/data
    depends_on:
      - redis                        # 可选：会话存储

  nginx:
    image: nginx:alpine
    container_name: cyberboss-nginx
    restart: unless-stopped
    ports:
      - '80:80'
      - '443:443'
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf
      - ./nginx/ssl:/etc/nginx/ssl
    depends_on:
      - cyberboss

  redis:
    image: redis:alpine
    container_name: cyberboss-redis
    restart: unless-stopped
    volumes:
      - redis_data:/data
```

**`Dockerfile`**：

```dockerfile
FROM node:20-slim

# Chromium 系统依赖
RUN apt-get update && apt-get install -y \
    ca-certificates fonts-liberation \
    libasound2 libatk-bridge2.0-0 libatk1.0-0 libcups2 \
    libdrm2 libgbm1 libgtk-3-0 libnspr4 libnss3 \
    libu2f-udev libxcomposite1 libxdamage1 libxfixes3 \
    libxkbcommon0 libxrandr2 xdg-utils \
    fonts-noto-cjk git \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app

# 安装 Claude Code CLI
RUN npm install -g @anthropic-ai/claude-code

# 项目文件
COPY package*.json ./
RUN npm ci --production

COPY src/ ./src/
COPY templates/ ./templates/
COPY public/ ./public/
COPY bin/ ./bin/
COPY scripts/ ./scripts/

# Puppeteer 自带 Chromium
ENV PUPPETEER_SKIP_DOWNLOAD=false

EXPOSE 3000
CMD ["node", "bin/server.js"]
```

> **🔴 决策 9.4：VPS 部署打包方式**
>
> - **方案 A（Docker Compose）**：一个 yml 文件定义全部服务，一键启动。环境一致，运维标准化。需要 VPS 装 Docker。
> - **方案 B（安装脚本 + PM2）**：`install.sh` 自动装依赖 → PM2 管理进程。更轻量，不需要 Docker。但环境一致性靠自己保证。
> - **方案 C（预构建虚拟机镜像）**：打包成 `.ova`/`.qcow2` 镜像。适合完全不懂技术的用户，但镜像大、更新不便。
>
> **建议**：方案 A（Docker Compose）。VPS 装 Docker 一行命令，Docker Compose 是服务部署的事实标准。

> **🟡 决策 9.5：Claude Code CLI 如何部署在容器内？**
>
> Claude Code 本身是一个 npm 全局包，可以在 Dockerfile 里 `npm install -g @anthropic-ai/claude-code`。但需要注意：
> - Claude Code 需要用户登录凭证（OAuth token）
> - 服务器端可以预配置 token 作为环境变量
>
> **建议**：Dockerfile 中安装 Claude Code CLI。每个用户的 Bot 进程使用各自的 Claude Code 配置（通过环境变量注入 API Key）。

> **🟡 决策 9.6：是否需要 CI/CD 自动部署？**
>
> - 手动：`git pull` + `docker compose up -d --build`
> - CI/CD：GitHub Actions → 推送到私有 Docker Registry → VPS 自动拉取
>
> **建议**：先手动部署。测试期频繁改动，手动更灵活。稳定后接入 GitHub Actions。
