## 1. 部署架构

### 1.1 目标

后端运行于服务器，用户通过前端网站扫码登录自己的 WeChat Bot，前端提供可视化配置界面。

### 1.2 架构方案

```
┌─────────────────────────────────────────────────────┐
│                    用户浏览器                          │
│         https://cyberboss.example.com                 │
│    ┌──────────┐  ┌──────────┐  ┌──────────────┐     │
│    │ 扫码登录  │  │ .env配置  │  │ 用量看板      │     │
│    └──────────┘  └──────────┘  └──────────────┘     │
└──────────────────────┬──────────────────────────────┘
                       │ HTTPS
┌──────────────────────▼──────────────────────────────┐
│                   服务器 (Node.js)                     │
│  ┌─────────┐ ┌──────────┐ ┌────────┐ ┌───────────┐ │
│  │ Web前端  │ │ Auth服务  │ │ Bot管理 │ │ 工单系统   │ │
│  ├─────────┤ ├──────────┤ ├────────┤ ├───────────┤ │
│  │ 静态文件  │ │ JWT/Sess │ │ 多Bot  │ │ 反馈收集   │ │
│  │ 服务     │ │          │ │ 生命周期│ │           │ │
│  └─────────┘ └──────────┘ └────────┘ └───────────┘ │
│                                                       │
│  ┌──────────────────────────────────────────────┐   │
│  │              MCP Tool Host                     │   │
│  │   (现有 cyberboss MCP tools 的 HTTP 包装)       │   │
│  └──────────────────────────────────────────────┘   │
└──────────────────────┬──────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          ▼            ▼            ▼
     ┌─────────┐ ┌─────────┐ ┌─────────┐
     │ Bot #1  │ │ Bot #2  │ │ Bot #N  │
     │ (WeChat)│ │ (WeChat)│ │ (WeChat)│
     └─────────┘ └─────────┘ └─────────┘
```

### 1.3 技术选型考量

| 层面 | 选项 A | 选项 B | 选项 C |
|------|--------|--------|--------|
| **前端框架** | React + Vite | Vue 3 + Vite | 纯 HTML + Vanilla JS |
| **后端扩展** | Express 嵌入现有 Node | 独立 Fastify 服务 | Next.js 全栈 |
| **扫码登录** | 微信机器人生成临时二维码 | 服务器生成 + WebSocket 轮询 | — |
| **多 Bot 管理** | PM2 多进程 | Docker 容器化 | 单进程多实例 |

### 1.4 待决策

> **🔴 决策 1.1：前端框架选择**
>
> - **React + Vite**：生态大，组件库丰富（Ant Design / shadcn），学习曲线适中。适合后续功能扩展。
> - **Vue 3 + Vite**：中文社区好，上手快（Naive UI / Element Plus），但部分复杂交互组件不如 React 生态成熟。
> - **纯 HTML + Alpine.js / HTMX**：最轻量，但没有组件化，维护成本随功能增长快速上升。
>
> **建议**：React + Vite + shadcn/ui（轻量、可定制、无 npm 依赖锁）。

> **🔴 决策 1.2：前端是否嵌入现有 Node 服务，还是独立部署？**
>
> - **嵌入**：在现有 `src/app/` 下加 Express 路由，单进程部署简单。但前后端耦合，前端改动需重启 Bot。
> - **独立**：单独前端服务 + API 网关。解耦干净，前端可独立扩缩。但增加运维复杂度。
>
> **建议**：独立服务 + 反向代理（Nginx），前端放在 `/web` 路由，API 放 `/api`。

> **🔴 决策 1.3：扫码登录方案**
>
> 微信 Bot 登录态如何转移到 Web？
> - **方案 A**：Bot 生成一次性 token → 用户发 token 到 Web → Web 拿到 token 调 API 建立 session。
> - **方案 B**：Web 展示二维码（含 sessionId）→ 用户在微信扫码 → Bot 回调服务器确认 → Web 轮询拿到登录态。
> - **方案 C**：预生成 API Key → 用户在 `.env` 里手动填入 → 前端直接用 API Key 认证。
>
> **建议**：方案 B（二维码）体验最好，但实现最复杂。方案 C 最简单但体验最差。**先落地 C + A 的混合方案**——预生成 API Key 作为初始认证，生成 token 提升体验。

## 2. 环境变量可视化配置

### 2.1 目标

用户在前端以**选项勾选**或**填空**形式配置模型与 API 密钥，填空项提供模板示例。

### 2.2 配置项清单

当前 `.env` 关键配置项：

```env
# === 必填 ===
CYBERBOSS_USER_NAME=            # 文本输入 + 示例：小方
CYBERBOSS_ALLOWED_USER_IDS=     # 文本输入 + 示例：wxid_xxxxxxxx
CYBERBOSS_WORKSPACE_ROOT=       # 路径选择器
CYBERBOSS_RUNTIME=              # 下拉：claudecode | openai | ...
CYBERBOSS_ACCOUNT_ID=           # 文本输入
CYBERBOSS_OBSIDIAN_VAULT=       # 路径选择器

# === 模型配置 ===
CYBERBOSS_MODEL_PROVIDER=       # 下拉：openai | anthropic | deepseek | ...
CYBERBOSS_MODEL_NAME=           # 下拉 + 自定义输入
CYBERBOSS_API_KEY=              # 密码输入 + 示例：sk-xxxx / ghp_xxxx
CYBERBOSS_API_BASE_URL=         # 文本输入 + 示例：https://api.openai.com/v1

# === 可选 ===
CYBERBOSS_DAILY_SUMMARY_TIME=   # 时间选择器 + 默认 22:57
CYBERBOSS_TZ=                   # 时区下拉
```

### 2.4 安全注意事项

- API Key 必须加密存储（AES-256-GCM），前端展示时脱敏
- `.env` 写入必须是服务端操作，不能通过前端直接写文件
- 配置变更需记录审计日志（谁、何时、改了哪个字段）

### 2.5 待决策

> **🔴 决策 2.1：配置存储方案**
>
> - **方案 A**：直接写 `.env` 文件（简单，但并发写有风险）
> - **方案 B**：SQLite 存配置 + 导出 `.env`（原子操作，可回滚，支持历史版本）
>
> **建议**：方案 B。SQLite 存配置元数据，写 `.env` 作为最终生效步骤。支持配置历史 + diff。

> **🔴 决策 2.2：模型提供商的选项来源**
>
> - **静态列表**：硬编码常见提供商（OpenAI / Anthropic / DeepSeek / 通义千问 / 智谱）
> - **动态模板**：维护一个 `provider-templates.json`，包含各提供商的默认 Base URL、模型列表
>
> **建议**：静态列表 + 动态模板。首次加载静态列表，后台从 GitHub raw 拉取最新模板更新。

## 3. 用户仓库管理

### 3.1 目标

用户的 Vault 仓库保存在本地（或同步至本地），服务器不存储用户私有数据。

### 3.2 方案分析

这是一个**核心架构约束**：服务器不持有用户数据。

```
用户本地机器                          服务器
┌─────────────┐                    ┌──────────┐
│ Obsidian    │  ←── MCP Tools ──  │ Bot 进程  │
│ Vault       │     (本地文件操作)   │          │
│             │                    │ 不存储    │
│ .env        │  ←── API 配置 ──   │ vault     │
│ 配置        │                    │ 内容      │
└─────────────┘                    └──────────┘
```

### 3.3 关键问题

1. **Bot 进程在哪里运行？** 如果 Bot 在服务器，它如何操作用户本地的 Obsidian Vault？
   - → Bot 必须在**用户本地**运行，服务器只做 Web 前端 + Auth + 转发
   - 或者：Bot 在服务器运行，但 vault 操作通过本地 Agent 完成

2. **多设备同步**：用户可能在手机看 Obsidian，电脑跑 Bot
   - → 建议用 Obsidian Sync / Git / Syncthing 做 vault 同步，Cyberboss 不负责

### 3.4 待决策

> **🔴 决策 3.1：Bot 部署位置**
>
> - **方案 A（本地模式）**：每个测试者在自己电脑跑 Bot + Claude Code。服务器只提供 Web 前端。用户数据完全本地。
> - **方案 B（服务器模式）**：Bot 在服务器运行，vault 通过 WebDAV/SSHFS 挂载到服务器。有数据安全风险。
> - **方案 C（混合模式）**：Bot 核心在本地，服务器只托管 Web 管理面板 + 用量统计聚合。
>
> **建议**：方案 A（本地模式）——符合"服务器不存储用户私有数据"的约束。Web 面板通过本地 localhost API 与 Bot 通信。

> **🔴 决策 3.2：Vault 备份策略**
>
> 如果 vault 在本地，测试期间数据丢了怎么办？
> - **方案 A**：不做自动备份，让用户自己用 Git/Obsidian Sync
> - **方案 B**：Bot 自动定时 `git commit + push` 到用户自己的私有仓库
> - **方案 C**：每天导出 vault 快照到加密 zip，可选上传到服务器
>
> **建议**：方案 B（自动 git 备份）。项目已支持 git 操作，加一个 cron 即可。

## 4. 报告同步与工单系统

### 4.1 目标

- 服务器仅同步**测试者的用户反馈**，不涉及普通用户的测试数据
- 增加工单系统，可直接返回给开发者工单

### 4.2 反馈流转

```
测试者微信 ──→ Bot ──→ 用户反馈 MCP Tool
                          │
                          ├─→ 测试者本地 vault (完整版)
                          │
                          └─→ 服务器工单系统 (脱敏版)
                                │
                                ├─→ 开发者查看
                                ├─→ 分配/标记状态
                                └─→ 回复 → 通知测试者
```

### 4.3 工单数据模型

```typescript
interface Ticket {
  id: string;                    // TK-20260626-001
  type: 'bug' | 'feature' | 'ux' | 'other';
  title: string;
  content: string;               // 原始反馈内容
  priority: 'high' | 'medium' | 'low';
  status: 'open' | 'in_progress' | 'resolved' | 'closed';
  reporter: string;              // 测试者标识（脱敏）
  assignee: string;              // 开发者
  tags: string[];
  createdAt: string;
  updatedAt: string;
  comments: Comment[];           // 开发者 ↔ 测试者对话
}
```

### 4.4 待决策

> **🔴 决策 4.1：工单系统实现方式**
>
> - **方案 A（自建）**：在服务器内置轻量工单系统（SQLite + 前端页面）。完全可控，但开发量大。
> - **方案 B（GitHub Issues）**：反馈自动创建 GitHub Issue。零开发前端，但测试者看不到进度，且 Issue 是公开的（或需加测试者到私有仓库）。
> - **方案 C（飞书/企微多维表格）**：反馈写入多维表格，开发者在飞书/企微查看。协作方便，但依赖第三方。
>
> **建议**：方案 A（自建轻量工单）。测试者需要看到自己的反馈状态，自建系统体验闭环。MVP 阶段可用一个反馈列表页 + 状态标记。

> **🔴 决策 4.2：反馈上报的触发方式**
>
> - **自动上报**：所有 `cyberboss_user_feedback` 自动同步到服务器
> - **手动上报**：测试者在反馈时选择"上报到开发者"
> - **混合模式**：标记为 `bug` 的自动上报，其他类型手动选择
>
> **建议**：混合模式。bug 自动上报（减少遗漏），feature/ux 手动选择（尊重隐私）。

> **🔴 决策 4.3：服务器与测试者 Bot 的通信协议**
>
> - **WebSocket**：长连接，实时双向。但需要 Bot 持续在线。
> - **HTTP Polling**：Bot 定时拉取服务器更新。简单但延迟高。
> - **Webhook**：Bot 暴露本地 HTTP endpoint。需内网穿透。
>
> **建议**：HTTP Polling（每 5 分钟拉一次）+ 手动"同步"按钮。MVP 不引入 WebSocket 复杂度。

### 5.3 待决策

> **🔴 决策 5.1：API 文档生成方式**
>
> - **方案 A（OpenAPI/Swagger）**：标准 `swagger.json` → Swagger UI 渲染。生态完善，支持在线测试。但需要额外写 schema。
> - **方案 B（手写文档页）**：写 Markdown + 代码高亮 + 复制按钮。灵活但维护成本高。
> - **方案 C（MCP Tool 自描述）**：利用现有 MCP Tool 的 inputSchema，自动生成文档。零额外维护。

> **🔴 决策 5.2：在线测试功能**
>
> API Explorer 里的"在线测试"按钮是否真的调 API？
> - **调真实 API**：直接调测试者本地 Bot 的 HTTP endpoint → 需要本地网络可达
> - **调 Mock**：展示假返回结果 → 安全但不够真实
>
> **建议**：先 Mock 展示（降低复杂度）。等本地 Agent 方案确定后再开真实调用。

## 6. API 用量统计

### 6.1 目标

前端提供**用量统计**看板，展示各 API 的调用次数、消耗配额等数据。

### 6.2 统计数据模型

```typescript
interface UsageMetric {
  date: string;                 // 2026-06-26
  provider: string;             // anthropic / openai / deepseek
  model: string;                // claude-opus-4-8
  totalCalls: number;           // 总调用次数
  totalTokens: {
    input: number;              // 输入 token
    output: number;             // 输出 token
    cacheRead: number;          // 缓存读取
    cacheWrite: number;         // 缓存写入
  };
  estimatedCost: number;        // 估算费用 (USD)
  bySkill: {                    // 按 Skill 拆分
    'flash-memory': { calls: number; tokens: number; cost: number };
    'commute-quiz': { calls: number; tokens: number; cost: number };
    'daily-summary': { calls: number; tokens: number; cost: number };
    // ...
  };
}
```

### 6.3 看板设计

```

│  ░░░░░  flash-memory (28%)                │

│  ░░     chat (12%)                        │
│  ░      other (7%)                        │
│                                            │

│   6/20  6/22  6/24  6/26                  │
```

### 6.4 实现路径

1. **数据采集**：在 Claude Code 调用后解析 usage metadata（`_claude_usage` 等环境变量或日志）
2. **本地存储**：SQLite 存每笔调用记录
3. **聚合展示**：前端通过 API 读取 SQLite 数据，Chart.js / Recharts 渲染

### 6.5 待决策

> **🔴 决策 6.1：用量数据的采集粒度**
>
> - **粗粒度（Skill 级）**：只记录每次 Skill 调用的总 token/cost。数据量小，够用。
> - **细粒度（请求级）**：记录每次 LLM API 请求。数据量大，但可以分析单次耗时、失败率。
>
> **建议**：粗粒度起步。Skill 级足以回答"哪个功能最烧钱"，且实现简单。

> **🔴 决策 6.2：用量数据是否上报到服务器？**
>
> - **不上报**：数据只在本地看板展示。完全隐私。
> - **匿名上报**：脱敏后上报聚合数据到服务器，用于全局对比（"你的用量在测试者中排前 20%"）。
> - **全量上报**：用于开发者分析系统行为。
>
> **建议**：本地展示为主 + 可选匿名上报（checkbox 默认不勾选）。

> **🔴 决策 6.3：可视化库选择**
>
> - **Recharts**（React）：声明式，组件化，和 shadcn/ui 搭配好
> - **Chart.js**：轻量，Canvas 渲染，与框架无关
> - **ECharts**：功能最全（热力图、地图），但包体积大
>
> **建议**：Recharts（假设前端选 React）。

### 7.3 打包方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. npm 包** | `npm i -g cyberboss-xiaofang` | 标准 Node 分发 | 需要 Node 环境 |
| **B. Docker Compose** | 一个 `docker-compose.yml` 拉起全部 | 环境一致，一键启动 | 需要 Docker |
| **C. 一键脚本** | `install.sh` / `install.ps1` 自动安装依赖 | 最轻量 | Windows/Mac 分开维护 |
| **D. 桌面安装包** | Electron 打包 `.exe` / `.dmg` | 用户体验最好 | 开发成本最高 |

### 7.4 推荐的 MVP 打包方案

```

├── install.ps1          # Windows 一键安装脚本
├── install.sh           # macOS/Linux 一键安装脚本
├── docker-compose.yml   # Docker 部署（可选）
├── start.bat            # Windows 启动脚本
├── start.sh             # macOS/Linux 启动脚本
├── .env.example         # 配置模板
└── README.md            # 测试者上手文档
```

### 7.5 待决策

> **🔴 决策 7.1：打包方式**
>
> - **方案 A（脚本 + npm）**：`git clone` + `npm install` + `npm start`。最简单，但需要测试者有一定的命令行基础。
> - **方案 B（Docker Compose）**：`docker compose up` 一键启动。环境隔离好，但需要测试者装 Docker。
> - **方案 C（Electron 桌面应用）**：下载 `.exe` 双击运行。体验最好，但开发周期长。
>
> **建议**：MVP 用方案 A（脚本 + npm）。测试者群体是开发者或技术用户，命令行不是障碍。后续可做 Docker 镜像。

> **🔴 决策 7.2：Web 管理面板的部署方式**
>
> 如果 Bot 在本地跑，Web 面板怎么访问？
> - **方案 A（localhost）**：面板在 `localhost:3456`，只能本机访问
> - **方案 B（内网）**：绑定 `0.0.0.0`，局域网设备可访问（手机看面板）
> - **方案 C（隧道）**：用 Cloudflare Tunnel / ngrok 暴露到公网
>
> **建议**：方案 A（localhost）起步。加一个 `--host 0.0.0.0` 参数支持内网访问。

> **🔴 决策 7.3：版本更新机制**
>
> - **手动**：`git pull` + `npm install`。简单，测试者可控。
> - **自动检查**：启动时检查 GitHub Release，提示有新版本。不强制更新。
> - **热更新**：Bot 自动拉取更新。有风险。
>
> **建议**：手动 + 自动检查（启动时 `git fetch` 并提示）。
