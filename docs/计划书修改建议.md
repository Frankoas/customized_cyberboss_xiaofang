#### AI 去重合并算法设计

```
输入：当日 inbox + 近期 uncategorized 闪存列表
处理流程：
  1. Embedding 向量化（调用 API 获取每条闪存的向量表示）
  2. 余弦相似度计算 → 相似度 > 0.85 → 标记为候选重复对
  3. AI 终判：将候选重复对发给模型判断"是否真的在说同一件事"
  4. 合并：语义相同 → 保留较早的，追加关联 ID
  5. 归档：标记为已处理，移出 inbox

输出：
  - 去重后的闪存列表
  - 合并报告（哪些被合并了）
  - 新发现的关联关系
```

#### 关联发现算法设计

```
输入：闪存条目
处理流程：
  1. 标签共现分析：两条闪存共享标签 → 关联度 +1
  2. 语义聚类：用 Embedding 将闪存聚为 N 个主题簇
  3. AI 确认：每个簇内的闪存是否确实相关
  4. 输出关联图谱：relatedFlashIds 双向关联

输出：更新后的闪存条目（含关联关系）
```

#### 空闲判定引擎逻辑

```
┌─────────────────────────────────────────────┐
│           统一空闲判定引擎                   │
├─────────────────────────────────────────────┤
│                                             │
│  日程更新事件                                │
│       ↓                                     │
│  查询当前日程 → 找到下一个 busy slot         │
│       ↓                                     │
│  计算闲暇窗口 = next_busy_start - now        │
│       ↓                                     │
│  生成闲散随机数（Beta 分布偏中期）            │
│       ↓                                     │
│  ┌─ 闲暇 < 0 ────→ 无空闲，跳过             │
│  ├─ 0 < 闲暇 < 30min → 碎片化记忆           │
│  ├─ 闲暇 > 30min → 大构思完善               │
│  └─ 用户手动触发 → 覆盖自动判定              │
│       ↓                                     │
│  开始前发送确认消息                          │
│  "现在有约{X}分钟空闲，想{刷题/完善构思}吗？" │
│       ↓                                     │
│  用户回复"好" → 开始                         │
│  用户回复"不要/忙/停" → 取消本次调度          │
│  用户 2 分钟未回复 → 再催一次 → 仍不回 → 取消│
│                                             │
└─────────────────────────────────────────────┘
```
